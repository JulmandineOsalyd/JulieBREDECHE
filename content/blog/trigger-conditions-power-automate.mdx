Il y a quelques mois, j'ai reçu une alerte : un flow chez un client venait de dépasser le quota mensuel d'exécutions. Le flow se déclenchait à chaque modification d'un élément SharePoint — y compris les modifications faites par le flow lui-même. Résultat : une boucle infinie, des milliers d'exécutions inutiles, et une facture de licences à revoir.

La solution ? Elle tenait en deux lignes. Elle s'appelle les **trigger conditions**.

---

## Le problème que personne ne vous dit au départ

Quand vous créez un flow Power Automate déclenché sur modification d'un élément SharePoint, d'un enregistrement Dataverse, ou d'une ligne Excel, le déclencheur s'active à **chaque modification** — sans exception.

C'est là que les ennuis commencent :

- Votre flow modifie un champ pour indiquer que le traitement est terminé → le flow se redéclenche
- Un utilisateur ouvre un élément et clique "Enregistrer" sans rien changer → le flow se redéclenche
- Un autre flow met à jour la même liste → les deux flows se redéclenchent mutuellement

Power Automate applique des **limites d'exécution** par abonnement (plan) et par utilisateur. Sur un plan standard, on parle de plusieurs milliers d'exécutions par mois — ce qui semble beaucoup jusqu'au jour où une boucle consomme tout ça en quelques heures.

Et sans les trigger conditions, la seule alternative est d'ajouter une condition en tout début de flow pour sortir immédiatement si les critères ne sont pas remplis. Le flow s'exécute quand même, consomme quand même une exécution, et vous avancez à reculons.

---

## Ce que font les trigger conditions

Les trigger conditions sont des expressions évaluées **avant** que le flow ne démarre. Si l'expression retourne `false`, le flow ne s'exécute tout simplement pas — aucune exécution comptabilisée, aucune action effectuée, aucun historique créé.

C'est une différence fondamentale avec une condition placée à l'intérieur du flow.

> **Retour terrain :** Chez un client avec 200 collaborateurs qui mettaient à jour une liste SharePoint toute la journée, on a réduit le nombre d'exécutions réelles de 90 % simplement en ajoutant une trigger condition sur le statut de l'élément. Le quota mensuel n'a plus jamais été atteint.

En pratique, les trigger conditions permettent de :

- **Éviter les boucles** : en filtrant les modifications faites par le compte de service du flow lui-même
- **Économiser les exécutions** : en ne déclenchant le flow que si un champ spécifique a changé
- **Cibler précisément** : en limitant le déclenchement à certaines valeurs, certains utilisateurs, ou certaines conditions métier

---

## Comment ajouter une trigger condition

Dans Power Automate, ouvrez votre flow et cliquez sur le déclencheur pour le développer. En haut à droite du panneau, cliquez sur les **trois points (…)** puis sur **Paramètres**.

Vous trouverez une section **Trigger Conditions**. Cliquez sur **Ajouter** pour saisir votre expression.

Les expressions utilisent la syntaxe des expressions Power Automate (la même que dans les formules dynamiques). Quelques exemples courants :

**Déclencher uniquement si le statut est "Approuvé" :**
```
@equals(triggerBody()?['Statut']?['Value'], 'Approuvé')
```

**Déclencher uniquement si un champ numérique dépasse un seuil :**
```
@greaterOrEquals(triggerBody()?['Montant'], 1000)
```

**Éviter une boucle — ne pas déclencher si la modification vient du compte de service :**
```
@not(equals(triggerBody()?['Editor']?['Email'], 'monflow@mondomaine.com'))
```

**Déclencher uniquement si le champ "Traité" est false :**
```
@equals(triggerBody()?['Traite'], false)
```

Vous pouvez ajouter **plusieurs trigger conditions** : elles sont toutes évaluées, et le flow ne démarre que si **toutes** retournent `true` (logique ET).

## L'astuce du Filter Array pour trouver la bonne syntaxe

Le point de friction avec les trigger conditions, c'est l'écriture de l'expression. Quelle est la bonne propriété ? Quel est le format exact de la valeur ? Une faute de frappe et le flow ne se déclenche plus jamais — sans erreur explicite.

Voici mon astuce : **utilisez une action Filter Array pour tester votre expression d'abord**.

### Pourquoi ça fonctionne

L'action **Filter Array** (Filtrer un tableau) utilise exactement la même syntaxe d'expression que les trigger conditions. Vous pouvez donc :

1. Créer une version temporaire du flow avec le déclencheur non filtré
2. Ajouter une action **Filter Array** sur `[triggerBody()]` (un tableau d'un seul élément)
3. Construire votre condition visuellement avec le mode avancé
4. Exécuter le flow manuellement, ouvrir l'historique, et **copier l'expression générée**
5. Coller cette expression dans la trigger condition

### Exemple concret

Vous voulez déclencher le flow uniquement si le champ "Département" vaut "Finance". Dans le Filter Array :

- Source : `[triggerBody()]`
- Condition (mode avancé) : construisez `item()?['Departement']` = `Finance`

Power Automate génère automatiquement l'expression. Vous la copiez, vous remplacez `item()` par `triggerBody()`, et vous la collez dans les paramètres du déclencheur.

> Cette méthode est particulièrement utile pour les champs de type Choix, Personne, ou Recherche (Lookup) dont la structure JSON n'est pas intuitive. Plutôt que de deviner, vous laissez Power Automate vous montrer le chemin exact.

---

## Ce que je retiens

- Mettez toujours une trigger condition dès qu'un flow se déclenche sur modification — c'est une bonne pratique systématique, pas une optimisation avancée
- Pour éviter les boucles, filtrez sur le modificateur (`Editor`) ou sur un champ d'état que vous gérez vous-même
- Utilisez le Filter Array pour valider votre syntaxe avant de l'appliquer au déclencheur — vous gagnerez un temps précieux
- Surveillez vos quotas d'exécution dans le centre d'administration Power Platform : les pics anormaux sont souvent le signe d'un flow sans trigger condition

Si vous n'avez pas encore l'habitude de les utiliser, prenez cinq minutes pour revoir vos flows existants. Il y a de bonnes chances que l'un d'entre eux déclenche plus qu'il ne devrait.
